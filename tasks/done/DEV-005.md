# DEV-005 — M11 Wave 5: `/seal` endpoint with full 16-step validation

## SECURITY-CRITICAL TASK

This is the most complex and security-sensitive endpoint in M11. Do NOT start until
DEV-001 through DEV-004 are all complete and their integration tests pass. The seal
handler calls into connections, nominations, and shares tables — all of which must
exist and be populated correctly.

Read **ARCH-010 §5 in its entirety** before writing any code. That section is the
**single implementation reference** for this endpoint. It supersedes the split
validation sequences in ARCH-003 §4 and ARCH-006 §5. Do not implement against those
sections directly.

## Endpoint to implement

| # | Method + Path | Description |
|---|---|---|
| 16 | `PUT /api/capsules/:id/seal` | Seal a capsule: validate all recipient keys, tlock fields (if present), Shamir config (if present), and the multi-path fallback rule; write all crypto columns atomically; advance capsule `shape` to `sealed`. |

## Full 16-step validation sequence (ARCH-010 §5.1)

Execute checks **in this exact order**. Return the specified error at the first failure.
Do NOT validate out of order. Do NOT write any DB rows if any check fails.

### Pre-conditions

```
[0] Authenticate caller: must be the capsule owner.
    → HTTP 403 { "error": "forbidden" }

[1] Load capsule: must exist and belong to the caller.
    → HTTP 404 { "error": "not_found" }

[2] Check capsule state: capsule.shape must be 'open'.
    → HTTP 409 { "error": "capsule_not_sealable",
                 "detail": "capsule is not in 'open' shape" }

[3] At least one entry in recipient_keys[] is required.
    → HTTP 422 { "error": "missing_recipient_keys",
                 "detail": "at least one recipient key is required" }
```

### Per-recipient validation (loop over each entry in recipient_keys[])

```
[4] Validate wrapped_capsule_key: must be a syntactically valid ASYMMETRIC envelope
    with alg_id = "capsule-ecdh-aes256gcm-v1".
    Use EnvelopeFormat.validateAsymmetric(wrappedCapsuleKey).
    → HTTP 422 { "error": "invalid_wrapped_capsule_key",
                 "detail": "recipient <connection_id>: invalid envelope" }

[5] If tlock fields are present in the request body (tlock != null):
    Validate wrapped_blinding_mask is present AND is a syntactically valid ASYMMETRIC
    envelope with alg_id = "capsule-ecdh-aes256gcm-v1" for EVERY recipient.
    → HTTP 422 { "error": "missing_wrapped_blinding_mask",
                 "detail": "recipient <connection_id>: tlock capsules require wrapped_blinding_mask" }

[6] Validate connection_id references an existing connections row owned by the capsule
    owner AND contact_user_id IS NOT NULL (must be a bound connection).
    → HTTP 422 { "error": "invalid_connection_id",
                 "detail": "connection <id> is not bound or does not belong to this user" }
```

### tlock-specific validation (only if request body has non-null `tlock` fields)

```
[7] Validate tlock.wrapped_key is non-null and non-empty.
    → HTTP 422 { "error": "missing_tlock_wrapped_key" }

[8] Call timeLockProvider.validate(TimeLockCiphertext(chain_id, round, wrapped_key_bytes)).
    If TLOCK_PROVIDER=disabled: reject immediately.
    → HTTP 422 { "error": "tlock_not_enabled",
                 "detail": "tlock provider is disabled on this server" }
    If validate() returns false:
    → HTTP 422 { "error": "tlock_blob_invalid",
                 "detail": "tlock blob failed structural validation" }

[9] Validate tlock.chain_id is a known chain.
    Stub: chain_id must equal TimeLockCiphertext.STUB_CHAIN_ID.
    → HTTP 422 { "error": "unknown_tlock_chain",
                 "detail": "chain_id is not recognised by the configured tlock provider" }

[10] Validate tlock_round timing: expected publish time must be <= capsule.unlock_at + 1 hour.
     For stub: expected_publish = STUB_GENESIS_UNIX + round * STUB_PERIOD_SECS.
     (STUB_GENESIS_UNIX = 1_700_000_000L, STUB_PERIOD_SECS = 3L)
     → HTTP 422 { "error": "tlock_round_mismatch",
                  "detail": "tlock round expected publish time is more than 1 hour after unlock_at" }

[11] Validate tlock.tlock_key_digest is present and decodes to exactly 32 bytes
     (after base64url decode).
     → HTTP 422 { "error": "missing_tlock_key_digest" }

[12] Validate SHA-256(tlock.dek_tlock) == tlock.tlock_key_digest.
     → HTTP 422 { "error": "tlock_digest_mismatch",
                  "detail": "SHA-256(dek_tlock) does not match tlock_key_digest" }
     CRITICAL: dek_tlock is key material. DO NOT log the dek_tlock value at any level.
     Log only: "tlock digest validation failed for capsule <id>" at WARN level.
```

### Shamir-specific validation (only if request body has non-null `shamir` fields)

```
[13] Validate shamir.threshold >= 1 and shamir.total_shares >= 1.
     Validate shamir.threshold <= shamir.total_shares.
     → HTTP 422 { "error": "invalid_shamir_config", "detail": "<specific constraint>" }

[14] Count executor_nominations rows with status='accepted' for this capsule owner.
     Count must be >= shamir.total_shares.
     → HTTP 422 { "error": "insufficient_accepted_nominations",
                  "detail": "need <N> accepted nominations; only <M> found" }
     NOTE: Actual share blobs are submitted AFTER sealing via POST /executor-shares.
     At sealing time the server ONLY validates that enough accepted nominations exist.
```

### Multi-path fallback rule (always checked, regardless of tlock/Shamir)

```
[15] Evaluate:
     let all_bound    = all recipient_keys entries have contact_user_id IS NOT NULL
                        (already confirmed at step [6])
     let has_tlock    = tlock fields are non-null in the request
     let has_executor = shamir fields are present AND shamir.total_shares >= 1

     If NOT all_bound AND NOT (has_tlock OR has_executor):
     → HTTP 422 { "error": "sealing_validation_failed",
                  "detail": "one or more recipients lack a sharing pubkey and no tlock or
                             executor fallback is configured" }
```

### Write phase (only reached if ALL checks [0]–[15] pass)

```
[16] In a SINGLE database transaction:
     a. Write wrapped_capsule_key, capsule_key_format to capsules row.
        (wrapped_capsule_key = first recipient_keys entry's wrapped_capsule_key;
         additional recipients go to capsule_recipient_keys)
     b. Write tlock columns (tlock_round, tlock_chain_id, tlock_wrapped_key,
        tlock_dek_tlock, tlock_key_digest) if tlock is present.
        CRITICAL: tlock_dek_tlock MUST NOT be logged at any level.
     c. Write shamir_threshold, shamir_total_shares to capsules row if shamir present.
     d. Upsert capsule_recipient_keys rows for each entry in recipient_keys[].
        For tlock capsules: include both wrapped_capsule_key and wrapped_blinding_mask.
        For non-tlock capsules: wrapped_capsule_key only; wrapped_blinding_mask = null.
     e. Set capsules.shape = 'sealed', capsules.sealed_at = now().
     Commit. If any step fails, rollback the ENTIRE transaction and return HTTP 500.
```

## Request body shape (from ARCH-003 §8)

```json
{
  "recipient_keys": [
    {
      "connection_id": "<uuid>",
      "wrapped_capsule_key": "<base64url>",
      "capsule_key_format": "capsule-ecdh-aes256gcm-v1",
      "wrapped_blinding_mask": "<base64url or null>"
    }
  ],
  "tlock": {
    "round": 1234567,
    "chain_id": "<hex string>",
    "wrapped_key": "<base64url: IBE-encrypt(DEK_client)>",
    "dek_tlock": "<base64url: 32 bytes — DEK XOR DEK_client>",
    "tlock_key_digest": "<base64url: SHA-256(DEK_tlock)>"
  },
  "shamir": {
    "threshold": 3,
    "total_shares": 5
  }
}
```
`tlock` and `shamir` are optional — omit or send null for non-tlock/non-Shamir capsules.

## TimeLockProvider implementation (also implement in this task)

This task requires implementing the `TimeLockProvider` interface and `StubTimeLockProvider`
class defined in ARCH-006. Place these in a new package:
`HeirloomsServer/src/main/kotlin/digital/heirlooms/server/crypto/tlock/`

### Files to create:
- `TimeLockProvider.kt` — the interface (copy verbatim from ARCH-006 §1)
- `TimeLockCiphertext.kt` — the data class (copy verbatim from ARCH-006 §1)
- `StubTimeLockProvider.kt` — stub implementation per ARCH-006 §2
- `TimeLockDecryptionException.kt` and `TimeLockValidationException.kt` per ARCH-006 §2.4

### StubTimeLockProvider spec (ARCH-006 §2):
- Constructor takes `stubSecret: ByteArray` (32 bytes from `TLOCK_STUB_SECRET` env var)
- Blob format: 64 bytes: `blob[0..31] = plaintextKey (DEK_client)` +
  `blob[32..63] = HMAC-SHA256(stubSecret, unlockAt_epoch_seconds || plaintextKey)`
- `seal()`: STUB_GENESIS_UNIX = 1_700_000_000L, STUB_PERIOD_SECS = 3L,
  chain_id always = `TimeLockCiphertext.STUB_CHAIN_ID`
- `validate()`: checks blob.size == 64, chainId == STUB_CHAIN_ID, round >= 1
- `decrypt()`: checks clock.instant() >= unlockAt (derived from round), verifies HMAC,
  returns plaintextKey or null if gate not open

### Configuration (AppConfig.kt — extend):
- `TLOCK_PROVIDER` env var: `stub`, `sidecar`, `disabled` (default: `disabled`)
- `TLOCK_STUB_SECRET` env var: required when `TLOCK_PROVIDER=stub`; fail fast at startup
  if absent with log message: `FATAL: TLOCK_PROVIDER=stub but TLOCK_STUB_SECRET is not set`

## Key constraints — invariants I-1 through I-8 (ARCH-010 §6)

All of the following invariants must hold. Violation of any is a security defect.

- **I-1:** `capsules.wrapped_capsule_key` ALWAYS wraps the full capsule `DEK`, not
  `DEK_client`. This is the iOS compatibility guarantee. No exception.
- **I-2:** `capsule_recipient_keys.wrapped_blinding_mask` is non-NULL ONLY on tlock
  capsules AND ALWAYS wraps `DEK_client`.
- **I-3:** Shamir shares (in `executor_shares`) are ALWAYS computed over `DEK`, not
  `DEK_client`. (Server doesn't compute shares — just validates nomination count.)
- **I-4:** `capsules.tlock_dek_tlock` and the `/tlock-key` response body must NEVER
  appear in any log, trace, or metric. Not at DEBUG level. Not in test logs.
- **I-5:** `tlock_key_digest = SHA-256(DEK_tlock)` is stored at sealing time and
  MUST be verified at `/tlock-key` delivery time. A mismatch is HTTP 500, not soft error.
- **I-6:** The server MUST NOT return `DEK_tlock` until BOTH: `TimeLockProvider.decrypt()`
  returns non-null AND `now() >= capsules.unlock_at`. (Enforced in DEV-006 `/tlock-key`.)
- **I-7:** The multi-path fallback rule (step [15]) MUST be enforced at sealing time.
  A deferred-pubkey connection as the sole unlock path is always rejected.
- **I-8:** The capsule state machine gate (step [2]: `shape = 'open'`) is checked BEFORE
  any crypto validation. Never allow sealing of an already-sealed or delivered capsule.

## Additional logging prohibitions

In the seal request handler:
- Do NOT log the `tlock.dek_tlock` field value from the request body.
- Do NOT log the `tlock.wrapped_key` content.
- DO log: "Seal request received for capsule <id> [tlock=<bool>, shamir=<bool>]" at INFO.
- On tlock digest mismatch (step [12]): log "tlock digest validation failed for capsule
  <id>" at WARN. Do not include the dek_tlock value.
- Configure HTTP request-logging middleware to redact the seal request body or exclude
  `tlock.dek_tlock` from access logs.

  - `decrypt()` returns null before unlockAt.

- Integration test via local stack (TLOCK_PROVIDER=stub):
  - Seal a tlock capsule end-to-end: create capsule, add recipient with shared key,

  - Seal a Shamir capsule (assert nominations count check).

### Bug fixed during implementation

`StubTimeLockProvider.seal()` initially used `unlockAt.epochSecond` for the HMAC input, but `decrypt()` derives the epoch from the round number (`epochSecondsForRound(round)`). These differ when `unlockAt` falls between period boundaries. Fixed by using `epochSecondsForRound(round)` in both `seal()` and `decrypt()`.
