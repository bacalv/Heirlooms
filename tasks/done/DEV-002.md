# DEV-002 — M11 Wave 1: Connections CRUD endpoints

## Context

DEV-001 must be complete and merged before starting this task. The V33 migration must
have applied, creating the `connections` and `executor_nominations` tables and backfilling
existing friend pairs.

Connections are the identity layer for capsule recipients, executor nominees, and Shamir
shareholders. These five endpoints expose basic CRUD on the `connections` table. They
are purely relational — no capsule crypto is involved. They must be fully working before
Wave 2 (executor nominations) or Wave 4 (share distribution) can begin, because both
require valid `connection_id` values.

Follow the existing server patterns. Model your implementation on the pattern used in
`CapsuleRoutes.kt` / `CapsuleService.kt` / `CapsuleRepository.kt` for route handler,
service, and repository layering. Add the new routes to `AppRoutes.kt`.

## Endpoints to implement

| # | Method + Path | Description |
|---|---|---|
| 1 | `GET /api/connections` | List all connections owned by the authenticated user (all roles). |
| 2 | `POST /api/connections` | Create a new connection. Supports bound connections (`contact_user_id`) and deferred-pubkey placeholders (`email`). |
| 3 | `PATCH /api/connections/:id` | Update a connection's `display_name`, `roles`, or cache a newly-observed `sharing_pubkey`. |
| 4 | `DELETE /api/connections/:id` | Remove a connection. **Must reject with HTTP 409** if any `executor_nominations` row exists for this connection with `status IN ('pending', 'accepted')`. |
| 5 | `GET /api/connections/:id` | Fetch a single connection by ID. Owner-only. Returns HTTP 404 if not found or not owned by caller. |

## Request / response shapes

### POST /api/connections — request body
```json
{
  "display_name": "Alice",
  "contact_user_id": "<uuid or null>",
  "email": "<string or null>",
  "roles": ["recipient"]
}
```
- At least one of `contact_user_id` or `email` must be non-null (server enforces).
- `sharing_pubkey` is NOT accepted in the creation body — it is derived server-side
  from `account_sharing_keys` when `contact_user_id` is provided, or NULL for placeholders.
- Return HTTP 409 if a connection already exists for `(owner_user_id, contact_user_id)`
  or `(owner_user_id, email)` (UNIQUE constraint violations).
- Return HTTP 201 with the created connection object.

### GET /api/connections — response
```json
{
  "connections": [
    {
      "id": "<uuid>",
      "display_name": "Alice",
      "contact_user_id": "<uuid or null>",
      "email": "<string or null>",
      "sharing_pubkey": "<base64url or null>",
      "roles": ["recipient"],
      "created_at": "2026-05-16T10:00:00Z",
      "updated_at": "2026-05-16T10:00:00Z"
    }
  ]
}
```

### PATCH /api/connections/:id — request body
```json
{
  "display_name": "Alice Smith",
  "roles": ["recipient", "executor"],
  "sharing_pubkey": "<base64url or null>"
}
```
All fields are optional. Apply only those present. Return HTTP 200 with the updated
connection object. Return HTTP 404 if not found or not owned by caller.

### DELETE /api/connections/:id — error response (HTTP 409)
```json
{
  "error": "active_nominations_exist",
  "detail": "connection has active executor nominations and cannot be deleted"
}
```

### GET /api/connections/:id — response
Same shape as one entry from the list response, wrapped in `{ "connection": { ... } }`.

## Implementation reference

- **ARCH-010 §1 (Wave 1)** — endpoint list and backfill note
- **ARCH-010 §2.3** — DELETE blocking condition (HTTP 409 with active nominations)
- **ARCH-004 §§1–3, 5** — connections schema, bound vs. deferred-pubkey logic, role labels

Key schema detail: `connections.sharing_pubkey` is populated at creation time for bound
connections by looking up `account_sharing_keys` for the given `contact_user_id`. The
PATCH endpoint allows a client to update `sharing_pubkey` when the contact later
generates or rotates their sharing keypair.

## Key constraints

- All endpoints require session authentication (`SessionAuthFilter` — follow the pattern
  in existing route files).
- `DELETE` must check for active nominations (`status IN ('pending', 'accepted')`) before
  deleting. Do not rely solely on FK constraints for this check — the 409 must be returned
  explicitly before any DELETE is attempted.
- Connections are always scoped to `owner_user_id = authenticated user`. Never return
  or mutate another user's connections.
- No capsule crypto or envelope validation belongs in this task.

## Acceptance criteria

  - Attempt to delete a connection that has a `pending` nomination — assert HTTP 409.
- No secrets (DEK values, tlock key material) logged at any level (not applicable here
  but maintain the codebase-wide policy).

### Key implementation decisions
- `sharing_pubkey` is looked up from `account_sharing_keys` on creation for bound connections; NULL for placeholders.
- DELETE checks `executor_nominations.status IN ('pending', 'accepted')` explicitly before deleting — does not rely on FK constraints.

- Conflict on UNIQUE constraint violations detected by message text (handles both Postgres SQLSTATE 23505 and RuntimeException wrappers).
- `connectionRepo` parameter in internal `buildApp` has a no-op default for backward compat with existing tests.
