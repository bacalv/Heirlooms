## Background

Raised during v0.54 wrap-up session (2026-05-16). Current environment model:

- **Test** (`test.api.heirlooms.digital`, `heirlooms-test` DB) — throw-away developer
  playground. Recreated freely. Smoke test accounts are disposable.
- **Production** (`api.heirlooms.digital`, `heirlooms` DB) — live system with real
  user data.

There is no intermediate environment that tests code against production-shaped data
before it reaches real users. This gap means migration safety, data-dependent bugs,
and realistic smoke test behaviour can only be validated in production.

## Proposed third environment: Staging

A pre-production environment that:
1. Receives promoted code **after** the test cycle passes on the test environment
2. Runs against an **anonymised snapshot** of the production database
3. Has a set of **seeded smoke test accounts** for the automated smoke test (TST-008)
4. Is rebuilt from a fresh prod snapshot before each release cycle

## Pipeline design

### Snapshot + anonymise
1. Take a Cloud SQL export of the production `heirlooms` database
2. Restore to a new staging DB (separate Cloud SQL instance or separate DB on existing instance)
3. Anonymise user-identifiable metadata:
   - Scramble `users.username` and `users.display_name` (e.g. `user_<hash>`)
   - Clear `user_sessions` (all sessions invalidated)
   - Clear any email fields if added in future
   - Vault content (GCS blobs, wrapped keys, ciphertext) is already opaque — no change needed
4. Seed known smoke test accounts (fresh E2EE setup, deterministic credentials)
5. Point staging Cloud Run services at the staging DB and staging GCS bucket

### Why E2EE simplifies anonymisation
Because all vault content is encrypted under user-specific keys, GCS blobs and
wrapped keys are already ciphertext — meaningless without device private keys. The
anonymisation script only needs to touch user metadata fields, not vault content.

### Seeded smoke test accounts
- Created fresh in the staging DB after each snapshot (not carried over from prod)
- Have known credentials and E2EE key material (generated by the seed script)
- Named consistently (e.g. `staging-smoke-a`, `staging-smoke-b`) for TST-008

## Infrastructure required

- Staging Cloud Run services: `heirlooms-server-staging`, `heirlooms-web-staging`
- Staging Cloud SQL database: `heirlooms-staging` (on existing `heirlooms-db` instance
  or a new instance)
- Staging GCS bucket: `heirlooms-uploads-staging` (separate from production — current
  test environment incorrectly shares `heirlooms-uploads`)
- Staging secrets in Secret Manager: separate API key, DB password
- Anonymisation script: `scripts/anonymise-snapshot.sh` or similar
- Seed script: `scripts/seed-staging-accounts.sh`

## Known open issues to resolve first

- **GCS bucket sharing**: the current test environment shares `heirlooms-uploads` with
  production. This must be fixed (separate bucket for test, separate for staging) before
  a staging environment makes sense. Fix this as a prerequisite.

## Timing

**Not urgent** — build this when:
- TST-008 automation is mature enough to run against it
- Production user base is large enough that migration risk justifies the investment
- The team is large enough that a formal pre-production gate adds value
