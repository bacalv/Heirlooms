# DEV-003 — M11 Waves 2+3: Executor nomination lifecycle + capsule recipient linking

## Context

DEV-001 (migrations) and DEV-002 (connections CRUD) must both be complete before
starting this task. Valid `connection_id` values are required for nominations; the
connections endpoints from DEV-002 must be functional so integration tests can set up
connections before nominating executors.

This task batches Wave 2 (6 executor nomination lifecycle endpoints) and Wave 3 (1
capsule recipient link endpoint) because Wave 3 is a single PATCH endpoint with low
complexity that shares the same connection-dependency as Wave 2, and batching keeps
DEV-004 (share distribution) unblocked immediately after this task.

The executor nomination lifecycle is the offer-accept-revoke state machine described
in ARCH-004 §4: `pending → accepted → (revoked)` or `pending → declined`. Nominations
are the prerequisite for Shamir share distribution (DEV-004) — `/seal` validates that
enough accepted nominations exist before sealing.

## Endpoints to implement

### Wave 2 — Executor nomination lifecycle

| # | Method + Path | Description |
|---|---|---|
| 6 | `POST /api/executor-nominations` | Owner extends a nomination offer. Creates a `pending` row in `executor_nominations`. Body: `{ connection_id, message? }`. |
| 7 | `GET /api/executor-nominations` | Owner lists all nominations they have issued (all states). |
| 8 | `GET /api/executor-nominations/received` | Nominee lists nominations extended to them (all states). |
| 9 | `POST /api/executor-nominations/:id/accept` | Nominee accepts. Moves `status` to `accepted`, sets `responded_at`. |
| 10 | `POST /api/executor-nominations/:id/decline` | Nominee declines. Moves `status` to `declined`, sets `responded_at`. |
| 11 | `POST /api/executor-nominations/:id/revoke` | Owner revokes an accepted or pending nomination. Moves `status` to `revoked`, sets `revoked_at`. No automatic share rotation in M11 (M13 scope). |

### Wave 3 — Capsule recipient linking

| # | Method + Path | Description |
|---|---|---|
| 12 | `PATCH /api/capsules/:id/recipients/:recipientId/link` | Link a `capsule_recipients` row to a `connections` row. Body: `{ connection_id }`. |

## Request / response shapes

### POST /api/executor-nominations — request body
```json
{
  "connection_id": "<uuid>",
  "message": "I'd like you to be my executor."
}
```
- `connection_id` must reference a connection owned by the authenticated user.
- Returns HTTP 201 with the created nomination object.
- Returns HTTP 404 if the connection does not exist or does not belong to the caller.
- Returns HTTP 409 if a `pending` or `accepted` nomination already exists for this
  connection (prevent duplicate active nominations).

### Nomination object (used in list and single responses)
```json
{
  "id": "<uuid>",
  "connection_id": "<uuid>",
  "owner_user_id": "<uuid>",
  "status": "pending",
  "offered_at": "2026-05-16T10:00:00Z",
  "responded_at": null,
  "revoked_at": null,
  "message": "I'd like you to be my executor."
}
```

### GET /api/executor-nominations — response
```json
{ "nominations": [ <nomination object>, ... ] }
```

### GET /api/executor-nominations/received — response
Same shape. Filtered to nominations where the contact in the linked `connections` row
matches the authenticated user (i.e. `connections.contact_user_id = current_user_id`).

### POST /api/executor-nominations/:id/accept|decline
- Accept/decline: nominee only (caller must be `connections.contact_user_id` for the
  nomination's `connection_id`). Returns HTTP 200 with updated nomination object.
- Returns HTTP 403 if caller is not the nominee.
- Returns HTTP 409 if nomination is not in `pending` state.

### POST /api/executor-nominations/:id/revoke
- Owner only (caller must be `executor_nominations.owner_user_id`). Returns HTTP 200.
- Returns HTTP 403 if caller is not the owner.
- Returns HTTP 409 if nomination is already `declined` or `revoked`.

### PATCH /api/capsules/:id/recipients/:recipientId/link — request body
```json
{ "connection_id": "<uuid>" }
```
Validation (ARCH-010 §2.2):
- `connection_id` must reference a `connections` row owned by the authenticated user.
- The `capsule_recipients` row (`recipientId`) must belong to a capsule owned by the
  authenticated user.
- A given `connection_id` may only be linked to one `capsule_recipients` row per capsule
  (uniqueness per capsule). Return HTTP 409 if already linked.
- Returns HTTP 200 with the updated `capsule_recipients` row on success.
- Returns HTTP 404 if the capsule, recipient row, or connection does not exist or does
  not belong to the caller.

## Key constraints

- Accept/decline endpoints are nominee-only; revoke is owner-only. Enforce this strictly
  with HTTP 403 responses, not 404.
- The nomination state machine must be enforced: you cannot accept an already-accepted
  nomination, decline a revoked nomination, etc. Use HTTP 409 for state machine violations.
- Wave 3 link endpoint: do not allow linking a `connection_id` that belongs to a
  different user than the capsule owner.
- No capsule crypto or envelope validation belongs in this task.
- In M11, revocation does NOT trigger automatic Shamir share rotation. Log a warning
  (`WARN: nomination <id> revoked — owner must manually redistribute shares if applicable`)
  but do not attempt any share cleanup.
- All endpoints require session authentication.

## Acceptance criteria

- All 7 endpoints return correct HTTP status codes per the spec above.
- Unit tests for all route handlers covering happy path, 403, 404, and 409 cases for
  each endpoint (mock the service layer).
- Unit tests for the nomination state machine in the service layer: assert that invalid
  transitions (accept an accepted nomination, revoke a declined one, etc.) produce the
  correct error.
- Integration test via local stack covering the full nomination lifecycle:
  1. Create two user accounts and a friendship.
  2. Create a connection for the owner.
  3. Owner creates a nomination → assert `pending`.
  4. Nominee accepts → assert `accepted`.
  5. Owner revokes → assert `revoked`.
  6. Create a new nomination for the same connection → assert a fresh `pending` is allowed.
- Integration test for the recipient link endpoint:
  1. Create a capsule with a free-text recipient.
  2. Create a connection.
  3. PATCH to link → assert `connection_id` is set.
  4. Attempt to link the same connection to a second recipient on the same capsule →
     assert HTTP 409.
- No secrets logged at any level.
