# DEV-006 — M11 Waves 6+7: `/tlock-key` delivery + read-path amendments

## Context

## Wave 6 — GET /api/capsules/:id/tlock-key

### Full gate logic and validation sequence (ARCH-010 §5.3)

Execute checks in this exact order:

```
[1] Authenticate caller: must be an authenticated recipient of this capsule
    (present in capsule_recipient_keys for this capsule_id).
    → HTTP 403 { "error": "not_a_recipient" }

[2] Load capsule: must exist and be sealed with non-null tlock fields
    (tlock_dek_tlock IS NOT NULL).
    → HTTP 404 { "error": "not_found" }

[3] Check TLOCK_PROVIDER != disabled.
    → HTTP 503 { "error": "tlock_not_enabled" }

[4] Check now() >= capsules.unlock_at.
    → HTTP 202 {
        "error": "tlock_gate_not_open",
        "detail": "unlock_at has not passed",
        "retry_after_seconds": <seconds until unlock_at>
      }

[5] Call timeLockProvider.decrypt(TimeLockCiphertext(chain_id, round, tlock_wrapped_key)).
    If null (gate not yet open):
    → HTTP 202 {
        "error": "tlock_gate_not_open",
        "detail": "round not yet published",
        "retry_after_seconds": 30
      }

[6] Verify SHA-256(capsules.tlock_dek_tlock) == capsules.tlock_key_digest.
    Mismatch:
    → HTTP 500 (log ERROR: "tlock tamper detected for capsule <id>";
                DO NOT log tlock_dek_tlock value)

[7] Return:
    HTTP 200 {
      "dek_tlock": "<base64url: capsules.tlock_dek_tlock>",
      "chain_id": "<hex>",
      "round": <bigint>
    }
```

### LOGGING PROHIBITION (CRITICAL — ARCH-010 §5.3, ARCH-006 §6.2)

The response body of this endpoint and the value of `capsules.tlock_dek_tlock` must
NEVER appear in:
- Application logs (SLF4J at any level, including DEBUG and TRACE)
- Access/request logs (any HTTP middleware)
- Distributed traces (OpenTelemetry, Stackdriver)
- Error reporting (Sentry, Crashlytics)

DO log (at INFO): capsule ID, caller user ID, HTTP status code, response latency.
DO NOT log the response body or any key material.

Configure HTTP request-logging middleware to redact the response body for this path
(`/api/capsules/*/tlock-key`). In integration tests assert the response body is present
and correct, but do not log it.

The same prohibition applies to `tlock_dek_tlock` in any DB query logs. Ensure
`LOG_LEVEL` is not set to trace PostgreSQL query results in the test environment.

## Wave 7 — GET /api/capsules/:id (amended)

Amend the existing capsule detail response in `CapsuleRepresentation.kt` to include
the M11 columns. Add these fields to the existing capsule JSON response:

```json
{
  "id": "...",
  "shape": "sealed",
  "...existing fields...",
  "wrapped_capsule_key": "<base64url or null>",
  "capsule_key_format": "<string or null>",
  "tlock_round": 1234567,
  "tlock_chain_id": "<hex string or null>",
  "tlock_wrapped_key": "<base64url or null>",
  "tlock_key_digest": "<base64url or null>",
  "shamir_threshold": 3,
  "shamir_total_shares": 5
}
```

Rules:
- Null fields may be omitted from the JSON or serialised as null — be consistent.
  Prefer omitting null fields to keep responses clean.
- `tlock_dek_tlock` is NOT included in this response. It is returned only via
  `/tlock-key` after gate validation.
- `wrapped_capsule_key` is returned to any authenticated user who can view the capsule
  (owner or recipient). iOS reads this field to decrypt without a `/tlock-key` call.
- `tlock_wrapped_key` (the IBE ciphertext) is safe to return — it is only decryptable
  after the drand round publishes, which is the intended time-lock property.

## Wave 7 — GET /api/capsule-recipient-keys/:capsuleId (new endpoint)

Return all `capsule_recipient_keys` rows for the given capsule. Used by Android/web
clients to retrieve their `wrapped_blinding_mask` for the blinded delivery path.

Authentication: caller must be either:
- The capsule owner (`capsules.user_id = current_user_id`), or
- An authenticated recipient (their `connection_id` is present in `capsule_recipient_keys`
  for this capsule and their user ID matches `connections.contact_user_id`).

Response:
```json
{
  "recipient_keys": [
    {
      "connection_id": "<uuid>",
      "wrapped_capsule_key": "<base64url>",
      "wrapped_blinding_mask": "<base64url or null>"
    }
  ]
}
```

Rules:
- Return all rows for owners (needed for audit and re-distribution).
- Return only the caller's own row for recipients.
- Return HTTP 403 if the caller is neither owner nor recipient.
- Return HTTP 404 if the capsule does not exist.
- `wrapped_blinding_mask` is null for non-tlock capsules. Omit or return null — be
  consistent with the choice made for the capsule detail endpoint above.

## Key constraints

- **Invariant I-4 (ARCH-010 §6):** `tlock_dek_tlock` must never appear in any log,
  trace, or metric. This is non-negotiable. Verify by reviewing every log statement
  in the new route handler and service before submitting.
- **Invariant I-5:** SHA-256(`tlock_dek_tlock`) == `tlock_key_digest` must be verified
  at step [6] before returning. A mismatch is HTTP 500.
- **Invariant I-6:** Both gates must be open — `unlock_at` passed (step [4]) AND
  `TimeLockProvider.decrypt()` non-null (step [5]). Neither gate alone is sufficient.
- The `GET /api/capsule-recipient-keys` endpoint must not leak other recipients' key
  material to non-owners. Recipients see only their own row.
- For the amended `GET /api/capsules/:id`: `tlock_dek_tlock` is never included in
  this response — only `tlock_key_digest` (the digest is safe; the key material is not).
