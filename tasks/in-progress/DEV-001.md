# DEV-001 — M11 Wave 0: V33 + V34 Flyway migrations (schema only)

## Context

M11 introduces a connections identity layer and new cryptographic columns on the
capsule model. Before any endpoint work can begin, two Flyway migrations must be
authored, applied, and verified against the test stack.

**Important:** The ARCH-010 brief refers to these as "V31" and "V32", but V31 and V32
are already taken in this repo (`V31__shared_plot_trellis_staging.sql` and
`V32__add_require_biometric_to_users.sql`). You must use **V33** for connections and
**V34** for the capsule crypto columns.

No endpoint code belongs in this task. All subsequent DEV-002 through DEV-006 tasks
depend on these migrations being correct and committed first.

## Migrations to implement

### V33__m11_connections.sql

Implements everything described as "V31" in ARCH-004 §5 and ARCH-010 §3.1:

1. Create the `connections` table (columns: `id`, `owner_user_id`, `contact_user_id`,
   `display_name`, `email`, `sharing_pubkey`, `roles`, `created_at`, `updated_at`).
   Include the two CHECK constraints and two UNIQUE constraints from ARCH-004 §5.
2. Create `executor_nominations` table (columns: `id`, `owner_user_id`, `connection_id`,
   `status`, `offered_at`, `responded_at`, `revoked_at`, `message`). Include the CHECK
   on `status IN ('pending', 'accepted', 'declined', 'revoked')`.
3. Run the backfill INSERT to populate `connections` from existing `friendships` rows.
   Use the exact UNION ALL + ON CONFLICT DO NOTHING statement from ARCH-004 §5.
4. `ALTER TABLE capsule_recipients ADD COLUMN connection_id UUID NULL REFERENCES connections(id) ON DELETE SET NULL`.
5. Create indexes:
   - `idx_connections_owner` on `connections(owner_user_id)`
   - `idx_connections_contact` on `connections(contact_user_id) WHERE contact_user_id IS NOT NULL`
   - `idx_executor_nominations_owner` on `executor_nominations(owner_user_id)`
   - `idx_executor_nominations_connection` on `executor_nominations(connection_id)`
   - `idx_capsule_recipients_connection` on `capsule_recipients(connection_id) WHERE connection_id IS NOT NULL`

### V34__m11_capsule_crypto.sql

Implements everything described as "V32" in ARCH-003 §2 and ARCH-010 §3.1:

1. `ALTER TABLE capsules` to add columns:
   - `wrapped_capsule_key BYTEA NULL`
   - `capsule_key_format TEXT NULL CHECK (capsule_key_format IN ('capsule-ecdh-aes256gcm-v1', 'tlock-bls12381-v1'))`
   - `tlock_round BIGINT NULL`
   - `tlock_chain_id TEXT NULL`
   - `tlock_wrapped_key BYTEA NULL`
   - `tlock_dek_tlock BYTEA NULL`
   - `tlock_key_digest BYTEA NULL`
   - `shamir_threshold SMALLINT NULL CHECK (shamir_threshold >= 1)`
   - `shamir_total_shares SMALLINT NULL CHECK (shamir_total_shares >= 1)`
   - constraint `shamir_threshold_lte_total CHECK (shamir_threshold IS NULL OR shamir_threshold <= shamir_total_shares)`
2. Create `capsule_recipient_keys` table (PK: `capsule_id, connection_id`; columns:
   `capsule_id UUID NOT NULL REFERENCES capsules(id) ON DELETE CASCADE`,
   `connection_id UUID NOT NULL REFERENCES connections(id)`,
   `wrapped_capsule_key BYTEA NOT NULL`,
   `capsule_key_format TEXT NOT NULL DEFAULT 'capsule-ecdh-aes256gcm-v1'`,
   `wrapped_blinding_mask BYTEA NULL`,
   `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`).
3. Create `executor_shares` table (PK: `id UUID DEFAULT gen_random_uuid()`; columns:
   `capsule_id UUID NULL REFERENCES capsules(id) ON DELETE CASCADE`,
   `nomination_id UUID NOT NULL REFERENCES executor_nominations(id)`,
   `share_index SMALLINT NOT NULL`,
   `wrapped_share BYTEA NOT NULL`,
   `share_format TEXT NOT NULL DEFAULT 'shamir-share-v1'`,
   `distributed_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`).
4. Create indexes:
   - `idx_executor_shares_capsule` on `executor_shares(capsule_id) WHERE capsule_id IS NOT NULL`
   - `idx_executor_shares_nomination` on `executor_shares(nomination_id)`

**V34 has a FK dependency on V33:** `capsule_recipient_keys` references `connections(id)`
and `executor_shares` references `executor_nominations(id)`. V33 must be applied before
V34 or V34 will fail at table creation.

## Implementation reference

- **ARCH-010 §3** — migration sequencing, rollback analysis, deployment gate
- **ARCH-010 §4** — V33 backfill correctness review (locking, atomicity, edge cases)
- **ARCH-004 §5** — full DDL for `connections` and `executor_nominations` tables
- **ARCH-003 §2** — full DDL for new `capsules` columns, `capsule_recipient_keys`, `executor_shares`

Pre-migration check from ARCH-010 §4.6: before testing against any data-bearing
instance verify that `account_sharing_keys` has at most one row per `user_id`:
```sql
SELECT user_id, COUNT(*) FROM account_sharing_keys GROUP BY user_id HAVING COUNT(*) > 1;
```

## Key constraints

- Both migrations must be additive only (nullable new columns, new tables). The old
  server binary must continue to function against the new schema during the deployment
  window — do not add NOT NULL columns without defaults or alter existing column types.
- V33 must precede V34 — apply both in a single Flyway run.
- Backfill INSERT must use `ON CONFLICT DO NOTHING` to be idempotent on re-run.
- Do not touch any endpoint code, route handlers, or service classes in this task.

## Acceptance criteria

- Both migration files apply cleanly against a fresh Testcontainers PostgreSQL instance
  (the existing `SchemaMigrationTest` covers this — extend it if necessary to assert
  that all new tables and columns exist).
- Unit test: assert V33 creates `connections`, `executor_nominations`, and adds
  `capsule_recipients.connection_id`.
- Unit test: assert V34 creates `capsule_recipient_keys`, `executor_shares`, and adds
  all nine new columns to `capsules`.
- Unit test: assert the backfill produces one `connections` row per direction for each
  existing `friendships` row (test with at least one seeded friendship pair).
- `SchemaMigrationTest` must pass without modification to its assertions (new assertions
  may be added; existing ones must not break).
- No secrets (DEK values, tlock key material) are referenced in this task at all.
