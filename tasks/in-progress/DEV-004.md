# DEV-004 — M11 Wave 4: Executor share distribution

## Context

DEV-001 (migrations), DEV-002 (connections CRUD), and DEV-003 (nominations) must all
be complete before starting this task. Accepted executor nominations must exist before
shares can be submitted.

These three endpoints handle the upload and retrieval of Shamir secret-sharing share
blobs. Shares are asymmetrically wrapped to executor sharing pubkeys — the server stores
opaque `capsule-ecdh-aes256gcm-v1` envelopes and never sees the plaintext share values.

**Critical ordering:** Shares are uploaded AFTER a successful `/seal` call (DEV-005).
At sealing time the server validates that enough accepted nominations exist
(`shamir_total_shares` count check), but the share blobs themselves are submitted in a
separate `POST /executor-shares` request after sealing succeeds. This task implements
all three share endpoints; DEV-005 (seal) will call into the nominations count query
but does NOT directly depend on the share-upload endpoint being implemented first.

## Endpoints to implement

| # | Method + Path | Description |
|---|---|---|
| 13 | `POST /api/capsules/:id/executor-shares` | Owner uploads all Shamir shares for a sealed Shamir capsule. |
| 14 | `GET /api/capsules/:id/executor-shares/mine` | Accepted executor fetches their own wrapped share for this capsule. |
| 15 | `GET /api/capsules/:id/executor-shares/collect` | Author-authenticated: collect all shares for this capsule (testing/recovery). Death-verification-gated collection is M13 scope. |

## Request / response shapes

### POST /api/capsules/:id/executor-shares — request body
```json
{
  "shares": [
    {
      "nomination_id": "<uuid>",
      "share_index": 1,
      "wrapped_share": "<base64url of capsule-ecdh-aes256gcm-v1 envelope>",
      "share_format": "shamir-share-v1"
    }
  ]
}
```

Full validation sequence (ARCH-010 §5.2):
1. Caller must be the capsule owner.
2. Capsule must be sealed (`shape = 'sealed'`) with `shamir_total_shares` set (not null).
3. `shares` array length must equal `capsules.shamir_total_shares`.
   → HTTP 422 `{ "error": "wrong_share_count", "detail": "expected <N>, got <M>" }`
4. `share_index` values must be unique, 1-based, covering `1..N` without gaps.
   → HTTP 422 `{ "error": "invalid_share_indices" }`
5. Each `nomination_id` must reference an `executor_nominations` row with
   `status = 'accepted'` and `owner_user_id` matching the capsule owner.
   → HTTP 422 `{ "error": "invalid_nomination_id", "detail": "<id>" }`
6. Each `wrapped_share` must be a syntactically valid ASYMMETRIC envelope with
   `alg_id = "capsule-ecdh-aes256gcm-v1"` — use `EnvelopeFormat.validateAsymmetric()`.
   → HTTP 422 `{ "error": "invalid_wrapped_share", "detail": "share <index>" }`
7. `share_format` must be `"shamir-share-v1"`.
   → HTTP 422 `{ "error": "invalid_share_format" }`
8. In a single transaction: insert all `executor_shares` rows. Return HTTP 200 `{}`.

### GET /api/capsules/:id/executor-shares/mine — response
```json
{
  "share": {
    "wrapped_share": "<base64url>",
    "share_format": "shamir-share-v1",
    "share_index": 2,
    "capsule_id": "<uuid>"
  }
}
```
- Caller must be an accepted executor for this capsule (i.e. an `executor_nominations`
  row with `status = 'accepted'` where `connections.contact_user_id = current_user_id`
  and `nomination_id` is present in `executor_shares` for this capsule).
- Returns HTTP 403 if caller is not an accepted executor for this capsule.
- Returns HTTP 404 if no share has been distributed yet (capsule sealed but shares not
  yet submitted via the POST endpoint).

### GET /api/capsules/:id/executor-shares/collect — response
```json
{
  "shares": [
    { "share_index": 1, "wrapped_share": "<base64url>", "nomination_id": "<uuid>" },
    { "share_index": 3, "wrapped_share": "<base64url>", "nomination_id": "<uuid>" }
  ],
  "threshold": 3,
  "total": 5
}
```
- In M11: caller must be the capsule author (authenticated as `capsules.user_id`).
- Death-verification-gated collection (executor-initiated quorum) is M13 scope — do
  not implement it now. Add a code comment referencing M13.
- Returns HTTP 403 if caller is not the capsule author.

## Key constraints

- The server stores only wrapped (opaque) share blobs. It never decrypts or inspects
  the plaintext 64-byte Shamir share structure.
- `EnvelopeFormat.validateAsymmetric()` is the correct validation call for
  `wrapped_share` — same method used for `wrapped_capsule_key` in the seal endpoint.
- Validate that `share_index` values are exactly `1..N` without gaps before inserting
  any rows. Do not rely on the DB to catch this — validate in the service layer and
  return a clean HTTP 422.
- All share submission validation must be done before any DB writes. If any share in
  the array is invalid, reject the entire batch with HTTP 422 and write nothing.
- No share plaintext values, DEK values, or tlock key material must be logged.

## Acceptance criteria

- All 3 endpoints return correct HTTP status codes and response shapes.
- Unit tests for the route handler covering: happy path upload, wrong share count (422),
  invalid envelope format (422), duplicate share indices (422), invalid nomination (422).
- Unit tests for the service-layer validation of share indices (1-based, no gaps, unique).
- Integration test via local stack:
  1. Create a capsule, nominations, seal the capsule with `shamir` config, then POST shares.
  2. Assert HTTP 200 on valid share upload.
  3. Assert each executor can call `/mine` and receive their wrapped share.
  4. Assert the capsule author can call `/collect` and receive all shares with correct
     `threshold` and `total`.
  5. Assert a non-executor cannot call `/mine` — expect HTTP 403.
- No secrets logged at any level.
