# SE Brief: M10 E3 — Shared Plots + E2EE

**Date:** 12 May 2026
**Milestone:** M10 — Shared Plots
**Increment:** E3 of 4
**Type:** Server + Web. No Android (E4).

---

## Goal

Introduce shared plots: collection plots visible to multiple users, with a per-plot
group key for E2EE. After E3, Bret can create a "Family" plot, invite Sadaar (from
the friends list) or an external person (via invite link), and items approved into
the plot are readable by all members — without the server ever seeing plaintext DEKs.
Members can add items and invite others. Only the owner can remove items or delete
the plot. Android adoption is E4.

---

## Migration: V25 full

Runs after V25a (E2 migration). Contains the member, invite, and plot key additions.

```sql
-- V25b__plot_members_invites.sql

-- Members of shared/public plots (including the owner with role = 'owner')
CREATE TABLE plot_members (
    plot_id          UUID        NOT NULL REFERENCES plots(id) ON DELETE CASCADE,
    user_id          UUID        NOT NULL REFERENCES users(id),
    role             TEXT        NOT NULL CHECK (role IN ('owner', 'member')),
    -- plot_key wrapped to this member's account sharing pubkey
    wrapped_plot_key BYTEA       NULL,
    plot_key_format  TEXT        NULL,    -- 'p256-ecdh-hkdf-aes256gcm-v1'
    joined_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (plot_id, user_id)
);

CREATE INDEX idx_plot_members_user ON plot_members(user_id);

-- Invite tokens for plot membership (fallback for non-friends)
CREATE TABLE plot_invites (
    id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    plot_id    UUID        NOT NULL REFERENCES plots(id) ON DELETE CASCADE,
    created_by UUID        NOT NULL REFERENCES users(id),
    token      TEXT        NOT NULL UNIQUE,
    used_by    UUID        NULL REFERENCES users(id),
    used_at    TIMESTAMPTZ NULL,
    expires_at TIMESTAMPTZ NOT NULL,     -- created_at + 48 hours
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_plot_invites_token ON plot_invites(token) WHERE used_at IS NULL;
```

## E2EE: per-plot group key

Every shared plot has a symmetric 256-bit `plot_key`. It is generated by the owner
at plot creation time (client-side, via `crypto.getRandomValues` / `KeyGen`). The
server never sees the raw key — only wrapped copies.

### Key storage

`plot_members.wrapped_plot_key`: the `plot_key` wrapped to that member's account
sharing pubkey using `p256-ecdh-hkdf-aes256gcm-v1` (same algorithm as M9 item sharing).
Each member (including the owner) has their own wrapped copy. A member without a
`wrapped_plot_key` row cannot decrypt any items.

### Adding a member

1. Inviter fetches own `wrapped_plot_key` from `GET /api/plots/:id/plot-key`.
2. Inviter unwraps it using own sharing private key → raw `plot_key`.
3. Inviter fetches new member's sharing pubkey from `GET /api/keys/sharing/:userId`.
4. Inviter wraps `plot_key` to new member's pubkey.
5. Inviter calls `POST /api/plots/:id/members` with `{ userId, wrappedPlotKey,
   plotKeyFormat }`. Server inserts `plot_members` row.

### Adding an item to a shared plot

Items added to a shared plot must have their DEKs re-wrapped under the `plot_key`.
The client (the item's owner) performs this:

1. Client unwraps item DEK (from own `uploads` row, using master key or sharing privkey).
2. Client fetches own `wrapped_plot_key` → unwraps → raw `plot_key`.
3. Client wraps item DEK under `plot_key` using AES-256-GCM → `wrapped_item_dek`.
4. Client wraps thumbnail DEK similarly → `wrapped_thumbnail_dek`.
5. Client posts to `POST /api/plots/:id/items` or `POST /api/plots/:id/staging/:uploadId/approve`
   with the wrapped DEKs.

New envelope format ID: **`plot-aes256gcm-v1`**. Register in `AlgorithmIds` alongside
existing IDs. Format: symmetric AES-256-GCM wrap (same as `master-aes256gcm-v1` but
keyed by the plot key rather than the master key).

### Reading an item from a shared plot

1. Client fetches `GET /api/plots/:id/plot-key` → receives own `wrapped_plot_key`.
2. Unwrap with own sharing privkey → raw `plot_key`.
3. Unwrap `plot_items.wrapped_item_dek` with `plot_key` → item DEK.
4. Unwrap `plot_items.wrapped_thumbnail_dek` with `plot_key` → thumbnail DEK.
5. Decrypt content and thumbnail normally.

The plot key is cached in memory for the duration of the session; do not request it
per-item.

### Key rotation

Deferred. When a member is removed (no UI in M10), the plot key should be rotated.
This is an M11+ concern. For M10, membership is additive only — no removal path
exposed in the API.

## Server: new endpoints

### Shared plot creation

`POST /api/plots` with `visibility: 'shared'` triggers additional server-side setup:

1. Insert plot row with `criteria = NULL, visibility = 'shared'`.
2. Insert `plot_members` row: `role = 'owner', wrapped_plot_key = :wrappedPlotKey,
   plot_key_format = :format`.

The request body for a shared plot:
```json
{
  "name": "Family",
  "visibility": "shared",
  "wrappedPlotKey": "<base64>",
  "plotKeyFormat": "p256-ecdh-hkdf-aes256gcm-v1"
}
```

Server validates: `wrapped_plot_key` and `plot_key_format` are required for
`visibility = 'shared'`. Private plots must not include them.

### Plot key

**`GET /api/plots/:id/plot-key`**
Returns the calling user's wrapped plot key.
```json
{
  "wrappedPlotKey": "<base64>",
  "plotKeyFormat": "p256-ecdh-hkdf-aes256gcm-v1"
}
```
Returns 404 if the user is not a member. Returns 403 if the plot is not a shared plot.

### Members

**`GET /api/plots/:id/members`**
List members. Returns `[{userId, displayName, username, role, joinedAt}]`.
Only accessible to existing members.

**`POST /api/plots/:id/members`** (friends list path)
Invite a friend directly by userId. The inviting member must be a member themselves.
The client must supply the new member's wrapped plot key (it fetched the friend's
sharing pubkey and wrapped the plot key client-side).

```json
{
  "userId": "uuid",
  "wrappedPlotKey": "<base64>",
  "plotKeyFormat": "p256-ecdh-hkdf-aes256gcm-v1"
}
```

Server validates: `userId` must be a friend of the inviting user (check `friendships`
table). Returns 400 if not friends; 409 if already a member.
Response 201.

**`POST /api/plots/:id/invites`** (invite link path)
Generate a 48-hour invite token for users who are not yet friends.
```json
{ "token": "...", "expiresAt": "..." }
```
The invite link is `https://heirlooms.digital/plots/join?token=:token`. On redemption,
the joining user fetches the plot's existing member's sharing pubkey (via
`GET /api/keys/sharing/me`) and the inviter wraps the plot key for them. Because the
inviter is not present at redemption time, the invite flow uses a two-step approach:

1. **Initiate:** inviter generates invite token. Token stored in `plot_invites` without
   a wrapped plot key.
2. **Redeem (step 1):** recipient calls `GET /api/plots/join-info?token=:token` →
   receives `{ plotName, plotId, inviterUserId, inviterSharingPubkey }`.
3. **Redeem (step 2):** recipient generates its own ephemeral key, calls
   `POST /api/plots/join` with body:
   ```json
   {
     "token": "...",
     "recipientSharingPubkey": "<base64>"
   }
   ```
   Server marks invite used, notifies inviter via a pending_plot_key_requests table
   (or simpler: recipient polls until inviter wraps the key).

**Simpler alternative for M10:** require the inviter to be online to complete the
join. The join flow is: inviter opens the "Pending invites" screen, sees the
waiting recipient, wraps the plot key for them, and confirms. Not fully async, but
workable for the user count at this stage. Fully async invite key delivery is an
M11+ refinement. Pick whichever approach fits implementation time; note the trade-off
in PROMPT_LOG.

### Approve endpoint (shared plot variant)

`POST /api/plots/:id/staging/:uploadId/approve` when `visibility = 'shared'`:
body MUST include `wrappedItemDek`, `itemDekFormat`, `wrappedThumbnailDek`,
`thumbnailDekFormat`. Server saves to `plot_items`. Returns 400 if any DEK field
is missing.

### Access control

Every shared plot endpoint checks `plot_members` for the requesting user:
- Not a member → 404 (privacy-preserving; don't reveal the plot exists).
- Member → proceed.
- Owner actions (delete plot, remove any item) → 403 if member role.
- Member actions (remove own item) → check `added_by = userId`.

## Web changes

### Shared plot creation

New option in the "New plot" flow: choose Private or Shared. Selecting Shared:
1. Generates `plot_key` client-side (`crypto.subtle.generateKey({name:'AES-GCM',length:256})`).
2. Exports as raw bytes.
3. Wraps to own sharing pubkey (same ECDH-HKDF path as M9 item sharing).
4. Posts to `POST /api/plots` with `visibility: 'shared'` and the wrapped key.

### Plot key in `vaultCrypto.js`

New functions:
- `generatePlotKey()` → raw 32-byte key.
- `wrapPlotKeyForMember(plotKeyBytes, memberSharingPubkeySpki)` → `{wrappedKey, format}`.
- `unwrapPlotKey(wrappedPlotKey, ownSharingPrivkey)` → raw plot key bytes.
- `wrapDekWithPlotKey(dekBytes, plotKeyBytes)` → `{wrappedDek, format: 'plot-aes256gcm-v1'}`.
- `unwrapDekWithPlotKey(wrappedDek, plotKeyBytes)` → raw DEK bytes.

Plot key cached in module-level `Map<plotId, plotKeyBytes>` for session duration.

### Invite member UI (`InviteMemberModal.jsx`)

Two tabs: **Friends** and **Invite link**.

Friends tab: friend picker (same component as M9 sharing). On select, wraps plot key
for the friend and calls `POST /api/plots/:id/members`.

Invite link tab: calls `POST /api/plots/:id/invites`, displays copyable link.
Notes the limitation: inviter must confirm the pending join manually (M10 constraint).

### Shared plot view

Shared plots appear in the Garden sidebar for all members (owner and members). Rendered
as a collection plot: items from `GET /api/plots/:id/items`. Decrypted using the plot
key flow. A small "shared" badge on the plot row in the sidebar.

Members can see who added each item (`added_by` resolved to display name).
Owner sees a "Manage members" entry in the plot gear menu.

### Staging for shared plots

Staging approval from the web requires the client to re-wrap the item DEK. The `approve`
action on a shared plot staging item:
1. Unwrap item DEK from own `uploads` record (master key or sharing privkey path).
2. Fetch plot key (cached or fetched via `GET /api/plots/:id/plot-key`).
3. Wrap item DEK + thumbnail DEK with plot key → include in approval body.

## What E3 does NOT include

- Plot key rotation on member removal (no member removal UI in M10).
- Public plots (visibility = 'public' reserved; enforce `requires_staging` from day one
  but no creation UI).
- Async invite key delivery (inviter must be online to confirm join in M10).
- Android (E4).

## Acceptance criteria

1. `./gradlew test` passes — all new tests green, no regressions.
2. Create a shared plot on web. Invite a friend. Both can browse the plot in the Garden.
3. Add an item to the shared plot from the owner's account. The member can see and
   decrypt it (verify in browser with two logged-in windows).
4. Staging approval on a shared plot: item with wrapped DEKs appears in `plot_items`.
5. Non-member cannot discover the shared plot (404 on all endpoints).
6. Member cannot remove another member's item (403).
