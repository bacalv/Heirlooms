Architecture

System Architecture

Three clients — Android app, web app, and a public share viewer — communicate with a single Kotlin/http4k server on Cloud Run. All media is stored as opaque encrypted blobs in GCS; the server never decrypts content. Clients upload and download directly via presigned GCS URLs.

Security Mechanism 1

Registration & Key Ceremony (First Device Setup)

New users arrive via a 48-hour invite token. The app derives separate auth and master-key material from the passphrase using Argon2id, generates a device keypair, and registers with the server. The master key never leaves the device in plaintext — the server stores only the wrapped (encrypted) copy.

Argon2id split: The 64-byte KDF output is split so the same passphrase produces both an auth credential (auth_key → auth_verifier) and a key-encryption key (master_key_seed) — the server learns neither the passphrase nor the master key.

auth_verifier: SHA-256(auth_key) is what the server stores. Even a full DB dump doesn't yield the passphrase or master key.

Fake salts: For unknown usernames, the challenge endpoint returns a deterministic fake salt (HMAC-SHA256 of server secret + username) — preventing username enumeration via timing.

Security Mechanism 2

Device Pairing (Android → Web Browser)

The server is a blind relay throughout pairing. The Android device wraps the master key to the browser's ephemeral public key; the server stores the wrapped blob but never holds the plaintext master key. The browser's private key is non-extractable from WebCrypto.

Server as blind relay: The server stores wrapped_master_key_for_web but cannot decrypt it — it only has the web pubkey, not the privkey.

Non-extractable WebCrypto key: The browser's private key cannot be read via JavaScript APIs, only used for the unwrap operation.

IDB persistence: On subsequent visits the browser re-validates the session token via GET /api/auth/me, then loads the IDB material to auto-unlock without re-pairing.

Security Mechanism 3

Upload Encryption (Client-Side, Server-Blind)

Every file gets a fresh random data encryption key (DEK). The DEK encrypts the content, thumbnail, and EXIF metadata separately. Only the wrapped (encrypted) DEK reaches the server. Media bytes travel from the client directly to GCS via presigned URLs — the server is never in the data path.

Per-file DEK: Each upload has its own random key. Compromise of one file's DEK affects only that file.

Nonce discipline: AES-GCM nonce reuse under the same key causes catastrophic plaintext disclosure. Each encryption operation generates a fresh 12-byte nonce stored inside the envelope.

EXIF in the encrypted blob: GPS coordinates, camera make/model, and lens data live inside the DEK-encrypted metadata blob — the server sees a taken_at timestamp (needed for sort/filter) but nothing else identifiable.

OOM guard (v0.43.0): Preview clip generation and the upload worker both catch Throwable, not Exception — OutOfMemoryError is an Error subclass and would otherwise silently crash the worker with no retry.

Security Mechanism 4

Item Sharing (DEK Re-wrap)

Sharing never copies the GCS blob. Instead, the sender unwraps the item's DEK locally, re-wraps it to the recipient's sharing public key, and the server creates a second upload record pointing at the same storage object. The recipient uses their sharing private key to decrypt. Friendship is verified server-side at both steps.

No blob copy: The same GCS object serves both sender and recipient. Storage cost is zero for the share; the compost guard ensures the blob isn't deleted while any live shared reference exists.

Friend gate at two points: The server checks friendship when the sender fetches the recipient's pubkey, and again when the share call lands — a compromised client can't bypass it by calling endpoints directly.

Per-user independence: The recipient's copy has its own rotation and tag state, independent of the sender's original.

Sharing keypair lazy init: The sharing keypair is generated on first vault unlock after the M9 upgrade (ensureSharingKey()) and stored server-side wrapped under the master key.
