Core

Vault & Encryption

End-to-end encryption
Shipped

Every photo and video is encrypted on-device before upload. The server stores only ciphertext. Staff with full database and bucket access see only encrypted bytes.

All platforms
Per-file data encryption keys (DEK)
Shipped

Each upload gets a fresh 32-byte random DEK. Compromise of one file's key affects only that file. DEKs are wrapped under the master key before leaving the device.

All platforms
AES-256-GCM content encryption
Shipped

Content bytes, thumbnail bytes, and EXIF metadata are each encrypted separately under the file's DEK. Fresh 12-byte random nonce per encryption operation.

All platforms
Versioned envelope format
Shipped

All encrypted blobs use a single binary format: version byte · algorithm ID · nonce · ciphertext · auth tag. Unknown algorithm IDs fail loudly. Enables crypto agility without format breaks.

All platforms
Encrypted EXIF metadata
Shipped

GPS coordinates, camera make/model, lens model, ISO, aperture, and focal length are stored encrypted under the file's DEK. The server receives only taken_at in plaintext (needed for sort/filter).

ServerAndroidWeb
Encrypted thumbnails
Shipped

Thumbnail bytes are encrypted under the same DEK as the content. Displayed only after client-side decryption. Server never holds a plaintext thumbnail.

All platforms
Server-side envelope validation (no decryption)
Shipped

On upload confirm, the server checks envelope version, algorithm ID recognisability, and minimum byte lengths. It never inspects plaintext content at any point.

Server
DEK rotation
Shipped

Per-file key rotation via PATCH /api/content/uploads/{id}/rotation. Old and new envelopes coexist during migration; rotation is incremental and file-by-file.

Upload

Android share sheet integration
Shipped

"My Heirlooms" appears in the Android share sheet for any photo or video. Works from any app that can share media (gallery, camera, files).

Android
In-app camera capture
Shipped

+ Photo and + Video buttons in the Garden. Fire OS compatibility: uses plain ACTION_VIDEO_CAPTURE intent (not CaptureVideo contract which requires EXTRA_OUTPUT).

Android
Direct-to-GCS upload (presigned URLs)
Shipped

Encrypted bytes travel from the device directly to Google Cloud Storage via presigned PUT URLs. The server is never in the media data path.

AndroidServer
WorkManager background jobs
Shipped

One WorkManager job per file. Jobs survive app backgrounding and device restart. Automatic retry on connectivity failure. Byte-level progress reported to the Upload Progress screen.

Android
OOM-safe upload worker
Shipped

Preview clip generation and the upload worker catch Throwable (not just Exception). OutOfMemoryError on low-RAM devices is handled gracefully — upload still completes without the preview clip.

Android
Encrypted video preview clips
Shipped

Short preview clips generated during upload; encrypted and stored alongside the full video. Best-effort — suppressed gracefully on low-memory devices.

Android
EXIF extraction on confirm
Shipped

Server extracts EXIF from the uploaded blob after the client calls confirm. Sensitive fields are returned for the client to re-encrypt; taken_at and non-sensitive fields stored plaintext.

Capsules

Time Capsules

Capsule creation
Shipped

Create a named capsule with a recipient (free text), unlock date (must be future), optional message, and one or more vault photos. Created in Open state — fully editable.

Web
Capsule state machine
Shipped

Four states: Open (editable) → Sealed (frozen) → Delivered. Also Cancelled from Open or Sealed. Invalid transitions rejected by the server.

ServerWeb
Message versioning
Shipped

Each edit to a capsule message creates a new version row. The server preserves the full edit history; clients display the current version.

Server
Sealing animation
Shipped

Wax-seal animation plays when the user seals a capsule. Visual state indicator (open / sealed / delivered) on the capsule list and detail views.

Web
Navigation guard
Shipped

Unsaved changes in the capsule form trigger a confirmation prompt before discarding. Prevents accidental loss of draft message or photo selection.

Web
"In N capsules" on photo detail
Shipped

Photo detail view shows a count of active capsule memberships. Prevents composting items that are in a capsule (server enforces this as a precondition).

Web
Cryptographic sealing (tlock + Shamir)
Planned M9+

Sealed will become a cryptographic state: per-capsule key wrapped to recipient's pubkey + tlock-encrypted against the drand round for the unlock date. Author cannot decrypt after sealing.

All platforms
Milestone delivery (scheduled send)
Planned M10

Delivery of a capsule on its unlock date via in-app notification and email. The feature that makes Heirlooms more than cloud storage.

All platforms
Posthumous delivery (executor-mediated)
Planned M11

Capsules conditioned on the user's death, unlocked by a quorum of nominated executors via Shamir's Secret Sharing. Heirlooms never holds enough key material to decrypt.

Friends & Sharing

Invite-based friendship
Shipped

Friendships are created automatically when a new user redeems an invite. Bidirectional. Backfilled from existing redeemed invites at M9 deploy.

Server
Friends list screen
Shipped

Burger → Friends shows all connected friends with their display name and username.

Android
Item sharing (DEK re-wrap)
Shipped

Share a photo or video to a friend. The sender re-wraps the DEK to the recipient's sharing pubkey locally. GCS blob is shared, not copied. Server verifies friendship at both the key-fetch and share-call steps.

AndroidServer
Account sharing keypair
Shipped

Each user has a P-256 keypair used as the target for DEK re-wrapping during sharing. Private key is wrapped under the master key and stored server-side. Lazily initialised on first vault unlock post-M9.

AndroidServer
Shared item attribution
Shipped

"Shared by [display name]" shown in photo detail for received items. Person-icon indicator on thumbnails. Per-user independent rotation and tag state.

Android
Compost GCS guard
Shipped

Background cleanup skips GCS blob deletion when any live (non-composted) shared copy references the same storage key. Prevents the sender's compost action from destroying the recipient's copy.

Security

Authentication

Argon2id passphrase KDF
Shipped

64-byte output, 64 MiB memory, 3 iterations, 1 thread. First 32 bytes → auth_key; last 32 bytes → master_key_seed. One passphrase produces both auth credentials and key-encryption material.

AndroidWeb
Challenge/response login
Shipped

Client fetches auth_salt, derives auth_key locally, sends SHA-256(auth_key). The passphrase is never transmitted. The server stores only SHA-256(auth_key) as auth_verifier.

AndroidWebServer
Session tokens
Shipped

32 random bytes per session. Only SHA-256(token) is stored in the database. Sessions expire 90 days from last_used_at and are refreshed on every authenticated request.

Server
Anti-enumeration fake salts
Shipped

The challenge endpoint returns a deterministic fake salt for unknown usernames (HMAC-SHA256 of server secret + username, truncated to 16 bytes). An attacker cannot determine whether a username exists.

Server
Per-user data isolation
Shipped

All DB queries for uploads, plots, capsules, and keys are scoped to the authenticated user's ID. Cross-user reads return 404 (not 403) to prevent probing. Verified by 23-test isolation suite.

Server
Invite-based registration
Shipped

New users must have a valid invite token (48-hour expiry). Invite is atomically redeemed; tokens cannot be replayed. The invite sender and recipient become friends automatically.

Server

Devices & Pairing

QR-based device pairing
Shipped

Android generates a QR code + 8-digit code. Browser scans or types the code, generates an ephemeral P-256 keypair, and receives the master key wrapped to its pubkey. Server is a blind relay throughout.

AndroidWebServer
Android Keystore device keypair
Shipped

Each Android device generates a P-256 keypair at registration. Private key is stored in Android Keystore and never exported. Used to wrap/unwrap the vault master key.

Android
Non-extractable WebCrypto key
Shipped

The browser's pairing private key is generated with extractable: false. It cannot be read by JavaScript — only used for the unwrap operation. Persisted to IndexedDB as a CryptoKey object.

Web
IDB pairing persistence
Shipped

After pairing, the browser persists the ephemeral keypair and wrapped master key to IndexedDB. On subsequent visits, auto-validates the session and auto-unlocks the vault without re-pairing.

Web
Invite QR generation (Android)
Shipped

Devices & Access screen generates a QR code encoding the invite URL. Also provides an Android share-sheet option to send the invite link directly.

Vault Recovery

Passphrase-wrapped server backup
Shipped

At registration, the master key is wrapped under the Argon2id-derived master_key_seed and stored server-side. Enables web login recovery without a separate recovery phrase.

Server
24-word recovery phrase
Documented

Designed as the "I don't trust anyone" escape hatch. Generated and shown once at first-device setup with required acknowledgement. Implementation deferred post-M8.

All platforms
Social recovery via Shamir shares
Planned M9+

Master key (or per-capsule key) split across nominated executors via Shamir's Secret Sharing. Threshold-of-N unlock replaces the recovery phrase as the recommended default when a connections graph exists.

Tags
Shipped

Free-text tags on uploads. Used for filtering in Explore and as a compost precondition guard. Tag validation enforced server-side (length, character set).

AndroidWebServer
Compost (soft delete)
Shipped

Items with no tags and no active capsule memberships can be composted. Held for 90 days; restorable within the window. Permanently deleted by background cleanup. Not a trash can — a considered removal.

AndroidWebServer
Restore from compost
Shipped

Within the 90-day window, composted items can be restored via the Compost view. Restored item reappears in Just Arrived.

WebServer
Context-aware photo detail navigation
Shipped

The ?from=garden|explore|compost parameter determines the negative-action button label and back destination. Negative-action buttons are visually separated from destructive actions.

Background Services

Pending blobs cleanup
Shipped

Scheduled job hard-deletes composted uploads after the 90-day window (GCS blob + DB row). Guards against deleting blobs with live shared references. Also purges expired sessions.

Server
EXIF extraction service
Shipped

Extracts EXIF from confirmed uploads. Sensitive fields returned to the client for re-encryption; taken_at stored plaintext. Recovery job re-processes any uploads where extraction failed.

Server
Diagnostic events

Per-user scoped diagnostic event log. POST /api/diag records events; GET /api/diag returns only the authenticated user's events. Used for debugging upload and sync issues.

Cryptographic Primitives

Content & thumbnail encryption
AES-256-GCM

Alg ID: aes256gcm-v1. 32-byte DEK, 12-byte random nonce, 16-byte auth tag. One DEK per file, one nonce per encryption op.

DEK wrap under master key
AES-256-GCM

Alg ID: master-aes256gcm-v1. Master key used directly as a 256-bit AES key. Same envelope format as content.

Master key wrap to device / sharing pubkey
P-256 ECDH-HKDF-AES-256-GCM

Alg ID: p256-ecdh-hkdf-aes256gcm-v1. Ephemeral P-256 keypair per wrap. HKDF-SHA256 as KDF. Used for device pairing and item sharing.

Master key wrap under passphrase
Argon2id-AES-256-GCM

Alg ID: argon2id-aes256gcm-v1. 64 MiB · 3 iter · 1 thread. 16-byte random salt. Parameters stored alongside blob. Used for passphrase recovery backup.

Auth key derivation
Argon2id (64-byte)

Same params as above. First 32 bytes = auth_key (hashed before sending). Last 32 bytes = master_key_seed (used only locally for recovery). One passphrase, two keys.

Auth verifier storage
SHA-256

Server stores SHA-256(auth_key), not the auth_key itself. Login sends SHA-256(auth_key); server compares to stored verifier.

Session token storage
SHA-256

32 random bytes issued to client. Only SHA-256(token) stored in DB. Leak of DB does not expose valid session tokens.

Anti-enumeration fake salt
HMAC-SHA-256

Unknown usernames receive HMAC-SHA256(server_secret, username) truncated to 16 bytes as a deterministic fake salt. Indistinguishable from real salts to an attacker.
