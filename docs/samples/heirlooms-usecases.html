Use Cases & User Journeys

Heirlooms · v0.45.5 · 12 May 2026

Actors
👤 Vault Owner

The person whose memories live in the vault. Operates the Android app as primary device. May also access via a paired web browser.

👥 Friend / Recipient

Someone with their own Heirlooms account, connected via the invite-and-friendship system. Can receive shared items.

🖥 Server

Stores encrypted blobs and wrapped keys. Enforces auth, friendship rules, and upload state transitions. Never decrypts content.

📦 GCS (Storage)

Stores the raw encrypted bytes of every photo and video. Accessed directly by clients via presigned URLs — the server is not in the media path.

Onboarding

Getting started

UC-01
Create Account
A new user joins Heirlooms via an invite link and sets up their encrypted vault on their Android device.
Actors
Vault OwnerFriend (invite sender)Server
Preconditions
  • Invite sender has an Heirlooms account and has generated an invite link
  • Invite link is less than 48 hours old
  • New user has the Android app installed (sideloaded APK)
User Journey
STEP 1
Receive invite link via message or email
STEP 2
Open the Android app; see the invite code entry screen
STEP 3
Enter a username, display name, and passphrase (×2 to confirm)
STEP 4
App generates device keypair + master key locally; registers with server
STEP 5
Land on Welcome screen, then Garden ("A garden begins with a single seed")
Main Flow (steps)
  1. User taps invite URL; app opens to InviteRedemptionScreen
  2. User enters invite code, username, display name, passphrase, confirm passphrase
  3. App derives auth_key and master_key_seed from passphrase via Argon2id (64 MiB, 3 iterations)
  4. App generates P-256 device keypair; wraps master key to device pubkey
  5. App wraps master key under passphrase as recovery backup
  6. App calls POST /api/auth/register with invite token, auth verifier, wrapped keys
  7. Server redeems invite, creates user + friendship, issues session token
  8. App stores session token in EncryptedSharedPreferences; device privkey in Android Keystore
Alternative Flows
  • Invite expired (410): Screen shows inline error; user must request a new invite
  • Username taken (409): Screen shows inline error; user picks a different username
  • Keyboard covers submit button (landscape): Column is scrollable with IME padding; button always reachable
Postconditions
  • User account and session exist in the server database
  • Master key wrapped to device pubkey stored server-side; recovery backup stored server-side
  • Bidirectional friendship created between new user and invite sender
  • System plot "Just Arrived" created for the new user
  • User can upload photos and videos immediately
UC-02
Pair a Web Browser
The vault owner gains read access to their vault from a desktop web browser without entering their passphrase.
Actors
Vault Owner (Android)Vault Owner (Browser)Server (relay only)
Preconditions
  • User has an active Android session
  • Browser is open at heirlooms.digital/pair or via login page link
User Journey
STEP 1
On Android: Burger → Devices & Access → "Show pairing code"
STEP 2
On browser: visit /pair, enter the 8-digit code
STEP 3
Browser displays QR code; Android scans it
STEP 4
Android wraps master key to browser's ephemeral pubkey, sends to server
STEP 5
Browser polls, receives wrapped key, unwraps — vault is open
Main Flow (steps)
  1. Android calls POST /api/auth/pairing/initiate; receives 8-digit code
  2. Browser generates ephemeral P-256 keypair (non-extractable WebCrypto)
  3. Browser calls POST /api/auth/pairing/qr with code + pubkey; receives session_id
  4. Browser displays QR encoding {session_id, code}
  5. Android scans QR; unwraps own master key from Android Keystore
  6. Android re-wraps master key to browser's pubkey; calls POST /api/auth/pairing/complete
  7. Browser polls GET /api/auth/pairing/status every second
  8. Browser receives session token + wrapped master key; unwraps with ephemeral privkey
  9. Browser persists keypair + wrapped key to IndexedDB; vault unlocked
Alternative Flows
  • Subsequent browser visits: Browser calls GET /api/auth/me on load; if valid, reads IDB material and auto-unlocks — no re-pairing needed
  • Session expired (401): IDB is cleared; user is redirected to login page
  • Login page path: User enters passphrase → Argon2id → POST /api/auth/login → vault unlocked via passphrase recovery key
Postconditions
  • Web session created; session token stored in browser localStorage
  • Master key material persisted to IndexedDB (survives tab close)
  • Browser can decrypt and display all vault content without further user input
UC-03
Recover Vault Access
The vault owner regains access to their vault after losing their device, using their passphrase.
Actors
Vault OwnerServer
Preconditions
  • User has a valid passphrase (set at registration)
  • User knows their username
  • Passphrase recovery blob was stored at registration
Main Flow (steps)
  1. User opens web app and navigates to login
  2. User enters username; browser calls POST /api/auth/challenge to retrieve auth_salt
  3. Browser derives auth_key + master_key_seed from passphrase + salt (Argon2id)
  4. Browser calls POST /api/auth/login with SHA-256(auth_key)
  5. Server verifies against stored auth_verifier; issues session token
  6. Browser calls GET /api/keys/recovery to fetch passphrase-wrapped master key blob
  7. Browser decrypts blob using master_key_seed → master key recovered
  8. Vault is accessible; user can then pair a new Android device
Alternative Flows
  • Unknown username: Server returns a deterministic fake salt — no enumeration possible; login will fail at the verifier step
  • Wrong passphrase: auth_verifier mismatch → 401; no hint given as to which field is wrong
  • Founding user migration (M8 deploy): One-time POST /api/auth/setup-existing path for users upgrading from the legacy API-key model
Postconditions
  • Active session established; vault accessible in browser
  • User can initiate a new device pairing from the web to re-establish Android access
Daily Use

Everyday workflows

UC-04
Upload a Photo or Video
The vault owner saves a photo or video into their encrypted vault from their Android device.
Actors
Vault Owner (Android)ServerGCS
Preconditions
  • User has an active Android session with vault unlocked
  • Photo or video is available in the device gallery or camera
User Journey
ENTRY A
Share sheet: long-press photo in gallery → "My Heirlooms"
ENTRY B
In Garden: tap + Photo or + Video button
STEP 1
App encrypts content, thumbnail, and EXIF locally
STEP 2
Encrypted bytes uploaded directly to GCS via presigned URL
STEP 3
Confirm sent to server; item appears in Just Arrived
Main Flow (steps)
  1. App calls POST /api/content/uploads/create; receives upload_id + two presigned PUT URLs
  2. App generates a fresh 32-byte DEK; encrypts content + thumbnail + EXIF metadata (AES-256-GCM)
  3. App wraps DEK under master key
  4. WorkManager job uploads encrypted content blob to GCS (presigned PUT)
  5. WorkManager job uploads encrypted thumbnail blob to GCS (presigned PUT)
  6. App calls POST /api/content/uploads/{id}/confirm with wrapped DEK + encrypted metadata
  7. Upload Progress screen shows byte-level progress; "Go to Garden" button appears on completion
  8. Item lands in "Just Arrived" plot with olive-branch arrival animation
Alternative Flows
  • Multiple files: One WorkManager job per file; all run concurrently; progress shown per-file
  • Upload interrupted: WorkManager retries automatically on next connectivity
  • Out of memory during preview clip (large video on low-RAM device): OOM caught as Throwable; upload still completes without preview clip
  • Camera permission not granted (Fire OS): App requests CAMERA permission at launch-time before opening camera intent
  • Fire OS video capture: Uses plain ACTION_VIDEO_CAPTURE intent (no EXTRA_OUTPUT) due to Fire OS incompatibility with pre-created URIs
Postconditions
  • Encrypted content and thumbnail stored in GCS; server holds only wrapped DEK and encrypted metadata
  • Item visible in the user's Just Arrived plot and browsable in Explore
  • Server never held plaintext bytes at any point in the flow
UC-05
Browse the Garden
The vault owner reviews and organises recently arrived items and their plot collections.
Actors
Vault Owner
Preconditions
  • User is logged in with vault unlocked (Android or Web)
  • At least one upload exists
Main Flow (steps)
  1. User opens Garden tab — sees "Just Arrived" plot at top, then user-defined plots below
  2. Just Arrived polls every 30 seconds; new items slide in with olive-branch animation
  3. User horizontally scrolls each plot row (4-column thumbnail grid, LazyRow)
  4. User taps a thumbnail to open Photo Detail
  5. User taps a plot header pencil icon to rename or delete the plot
  6. User taps "+ Add plot" at the bottom of the garden scroll to create a new plot
  7. User long-presses a plot header for drag-to-reorder
Alternative Flows
  • Empty garden: "A garden begins with a single seed" empty-state illustration
  • Received shared item: Appears in Just Arrived with a person-icon indicator on the thumbnail
  • Thumbnail not yet available: Olive branch icon shown (not a spinner) during load and on failure
Postconditions
  • User has reviewed new arrivals; items are visible in their appropriate plots
UC-06
Explore the Vault
The vault owner browses their full collection with filtering and sorting — a leisure surface distinct from the work surface of the Garden.
Actors
Vault Owner
Main Flow (steps)
  1. User opens Explore tab — sees a 4-column paginated grid of all uploads, newest first
  2. User opens filter sheet to filter by tag, media type, or date range
  3. User changes sort order (newest first, oldest first, recently viewed)
  4. Grid loads more items as user scrolls (cursor-based pagination)
  5. User taps a thumbnail to open Photo Detail in Explore flavour (negative-action button is "Back to Explore")
  6. Server records a last_viewed_at timestamp per view for the "recently viewed" sort
Alternative Flows
  • No results for filter: Empty state shown; filter chrome remains visible
  • Composted items: Not shown in Explore; accessible only via the Compost view
UC-07
View a Photo or Video
The vault owner opens and views a specific encrypted item in full detail.
Actors
Vault OwnerGCS
Main Flow (steps)
  1. User taps a thumbnail from Garden, Explore, or Capsule
  2. App fetches presigned GET URL for the encrypted content blob from GCS
  3. App unwraps DEK using master key (or sharing privkey for received items)
  4. App decrypts content bytes using DEK (AES-256-GCM)
  5. Photo is displayed full-screen; video begins streaming
  6. Detail view shows: taken_at, tags, "in N capsules" count, rotation, "Shared by [name]" if applicable
  7. User can add/edit tags, rotate, add to capsule, share (if owned + encrypted), or compost from this screen
Alternative Flows
  • Large video on web: MSE streaming — video is fetched and decrypted chunk by chunk using mp4box; playback begins before the full file is downloaded
  • Received (shared) item: DEK unwrapped using sharing_privkey (p256-ecdh-hkdf-aes256gcm-v1 format detected automatically); "Shared by [Name]" shown
  • ?from= parameter: Determines the negative-action button label and destination (Garden / Explore / Compost)
Social

Friends & sharing

UC-08
Share an Item with a Friend
The vault owner sends an encrypted photo or video to a friend. The GCS blob is not copied — only the key wrapping changes.
Actors
Vault Owner (Sender)Friend (Recipient)Server
Preconditions
  • Sender owns the item (not a received share) and it is encrypted
  • Sender and recipient are connected as friends (via invite redemption)
  • Recipient has completed their sharing keypair setup (lazy init on first vault unlock post-M9)
User Journey
STEP 1
Tap share icon (bottom-right) on an owned encrypted thumbnail
STEP 2
Friend picker sheet opens; select recipient
STEP 3
App fetches recipient's sharing pubkey; re-wraps DEK locally
STEP 4
Server creates recipient upload record (same GCS blob)
STEP 5
Item lands in recipient's Just Arrived
Main Flow (steps)
  1. User taps share icon (visible only on encrypted + owned items)
  2. App fetches friend list; user selects a friend
  3. App calls GET /api/keys/sharing/{recipientUserId} — server checks friendship
  4. App fetches upload metadata to get wrapped_dek
  5. App unwraps DEK using own master key; re-wraps to recipient's P-256 sharing pubkey (ECDH-HKDF-AES-GCM)
  6. App calls POST /api/content/uploads/{id}/share with wrapped DEKs for recipient
  7. Server verifies ownership + friendship (second check); creates recipient upload row
Alternative Flows
  • No friends yet: Friend picker shows empty state; share icon is still visible for when friends are added
  • Item already shared with this recipient: Server will create a second record (re-sharing is allowed)
Postconditions
  • Recipient has a new upload record with their own wrapped DEK; GCS blob is shared, not copied
  • Recipient's copy has independent rotation and tag state
  • GCS blob cannot be deleted by the sender's compost action while the recipient's copy is live
UC-09
Receive a Shared Item
A friend's photo appears in the recipient's vault, decryptable only with the recipient's sharing private key.
Actors
Friend (Recipient)ServerGCS
Main Flow (steps)
  1. Garden's 30-second Just Arrived poll returns the new shared item
  2. Item has thumbnail_dek_format = "p256-ecdh-hkdf-aes256gcm-v1"
  3. App detects non-standard format; fetches sharing_privkey (wrapped under master key) from server
  4. App unwraps sharing_privkey; uses it to unwrap the thumbnail DEK via ECDH-HKDF-AES-GCM
  5. Thumbnail decrypted and displayed with person-icon indicator
  6. Full item view shows "Shared by [Sender's display name]"
  7. Recipient can add tags, rotate, and manage the item independently
Alternative Flows
  • Arrival animation: Olive-branch arrival animation plays when item first appears in Just Arrived
  • Recipient composting their copy: Compost guard checks for other live references before deleting the GCS blob
Capsules

Time-locked memories

UC-10
Create a Time Capsule
The vault owner plants a collection of photos and a message to be delivered at a future date.
Actors
Vault OwnerServer
Preconditions
  • User is logged in; vault is unlocked
  • At least one upload exists to add to the capsule (optional for draft)
User Journey
STEP 1
Capsules tab → Create capsule
STEP 2
Enter recipient name (free-text), unlock date, optional message
STEP 3
Add photos from the vault via photo picker modal
STEP 4
Submit — capsule created in Open state
STEP 5
Continue editing until ready to seal
Main Flow (steps)
  1. User navigates to Capsules tab; taps Create
  2. User enters: recipient (free text), unlock date (must be future), message (optional)
  3. User opens photo picker modal; selects photos from their vault
  4. App calls POST /api/capsules; server creates capsule in Open state
  5. App calls POST /api/capsules/{id}/photos to add selected photos
  6. Capsule appears in list with visual "open" state indicator (wax seal open)
  7. User may return to edit recipient, message, photos, or unlock date while capsule is Open
Capsule States
● Open — editable ● Sealed — locked ● Delivered ● Cancelled
Alternative Flows
  • Unlock date in the past: Validation error shown; submission blocked
  • No recipient entered: Validation error shown; submission blocked
  • Cancel capsule: Capsule transitions to Cancelled; photos are unlinked but not deleted from vault
UC-11
Seal a Capsule
The vault owner freezes a capsule's contents. After sealing, neither the author nor Heirlooms can change what is inside.
Actors
Vault OwnerServer
Preconditions
  • Capsule is in Open state
  • User has reviewed the contents and is ready to commit
Main Flow (steps)
  1. User opens a capsule in Open state; taps "Seal capsule"
  2. Sealing animation plays (wax seal closing)
  3. User confirms via a confirmation dialog (sealing cannot be undone)
  4. App calls POST /api/capsules/{id}/seal
  5. Server validates state transition (Open → Sealed only); updates state
  6. Capsule now shows Sealed indicator; edit controls are hidden
Alternative Flows
  • Already sealed: Server rejects the transition with 409; client shows error
  • Navigation guard: If user navigates away mid-form with unsaved changes, a confirmation prompt is shown before discarding
  • Future — cryptographic sealing (M9+): Sealed will become a cryptographic state where even the author cannot decrypt; current sealing is a database-flag state only
Postconditions
  • Capsule state is Sealed; contents are frozen at this version
  • A new message version is recorded with a sealed_at timestamp
  • Capsule cannot be edited or unsealed by any party
Management

Organisation & curation

UC-12
Organise Items into Plots
The vault owner groups photos and videos into named plots for the Garden work surface.
Actors
Vault OwnerServer
Main Flow (steps)
  1. User scrolls to the bottom of the Garden; taps "+ Add plot"
  2. PlotCreateSheet opens; user enters a plot name
  3. App calls POST /api/plots; new plot appears at the bottom of the Garden
  4. User tags individual items from Photo Detail to associate them with the plot
  5. User taps the pencil icon on a plot header to rename or delete the plot
  6. User long-presses a plot row handle and drags to reorder
  7. App calls PATCH /api/plots with new display order
Alternative Flows
  • Delete plot: Confirmation dialog shown; plot deleted; items remain in vault (unassigned from that plot)
  • System plot (Just Arrived): Cannot be renamed, deleted, or reordered — always shown first
  • Web drag-and-drop reorder: Handled via DnD in the web Garden; same PATCH /api/plots endpoint
UC-13
Compost an Item
The vault owner soft-deletes a photo or video. It is held for 90 days before permanent removal — composting is considered, not casual.
Actors
Vault OwnerServer
Preconditions
  • Item has no tags assigned
  • Item is not a member of any active (non-cancelled) capsule
Main Flow (steps)
  1. User opens Photo Detail; taps "Compost" (negative-action button)
  2. Confirmation dialog explains the 90-day window
  3. App calls POST /api/content/uploads/{id}/compost
  4. Server checks: no tags, no active capsule memberships
  5. Item moves to Compost view; disappears from Garden and Explore
  6. User can restore the item within 90 days via the Compost view
  7. After 90 days: background cleanup service permanently deletes DB row and GCS blob
Alternative Flows
  • Item has tags: Server returns 409; user must remove tags before composting
  • Item is in an active capsule: Server returns 409; user must remove from capsule first
  • Item has live shared copies: GCS blob deletion is skipped by the cleanup service; only the sender's DB row is deleted
  • Restore: User opens Compost view; taps Restore; item returns to Garden in Just Arrived
Postconditions
  • Item is hidden from all browsing surfaces but recoverable for 90 days
  • After 90 days: DB row and GCS blob permanently deleted (GCS deletion guarded against live shared references)
